  • 什么是mybatis的缓存

    • 开启一级缓存sqlsession,在当前sqlsession中,执行相同查询不要要再次执行sql获取结果集,不同sqlsession中仍然需要
    • 开启二级缓存sqlsessionfactory,同属于一个sqlsessionfactory的sqlsession执行查询中不需要再次执行sql获取结果集,需要注意,一级缓存默认开启,二级缓存打开则需要<cached>进行配置,一级缓存不需要pojo实现系列化接口,二级缓存需要pojo实现序列化接口

  • 怎样开启自定义缓存

    • 开启自定义缓存,实现类需要实现接口mybatis的接口org.apache.ibatis.cacheCache.Cache接口
      • 获取缓存 String getId()
      • 保存对象 void putObject(Object key,Object value)
      • 获取缓存数据 Object getObject()
      • 删除缓存key为键 Object removeObject(Object key)
      • 清除缓存 void clear()
      • 获取读写锁,需要考虑线程场景 getReadWriteLock()

  • mybatis框架的运行过程是怎样的

    • 读取配置文件缓存到configuration对象中,用来创建SqlSessionFactory
    • sqlsession的执行过程

  • SqlsessionFactoryBuilder是怎样构建SqlsessionFactory的

    • 通过org.apache.ibatis.builder.xml.XmlConfigBuilder解析配置的xml文件,读出参数内容后存入org.apache.ibatis.session.Configuration类中,Configuration类采取单例模式,几乎所有的mybatis配置都会存储在这个单例对象中
    • 使用Configuration去创建SqlsessionFactory, SqlsessionFactory是一个接口,具体的实现类是org.apache.ibatis.session.DefaultSqlsessionFactory

  • 构建Configuration,Configuration有什么作用?

    • 读入配置文件,包括基础配置xml和映射器xml
    • 初始化一些基础配置,比如mybatis的别名,一些重要的类对象(插件,映射器,objectfactory工厂,typehandlers)
    • 提供单例,为后续创建sqlsessionfactory服务,提供配置的参数
    • xml==>Configuration==>SqlsessionFactory==>Sqlsession==>Mapper
      • Configuration是通过XMLConfigBuilder去构建的,首先它会读出所有xml配置的信息,然后把他们解析并保存在configuration单例中,它会作如下初始化
        • Properties 全局参数
        • typeAlias 别名
        • plugins插件
        • objectfactory工厂
        • objectwrapperfactory对象包装工厂
        • reflectionFactory反射工厂
        • settings 环境设置
        • environments 数据库环境
        • databaseIdProvider 数据库标识
        • typehandlers类型转换器
        • Mappers 映射器

SpringIOC容器简述

  • SpringIOC容器的设计
    • SpringIOC容器的设计是基于BeanFactory和ApplicationContext两个接口
    • BeanFactory是springIOC容器定义的最底层的接口,而applicationContext是其高级接口之一
    • BeanFactory接口中包含的抽象方法有
      • getBean
      • isSingleton(isProtoType)
      • isTypeMatch
      • getAlias
    • 接口对应关系
      BeanFactory==>ApplicationContext==>ClassPathXmlApplicationContext
    • 初始化Spring IOC容器的形式
      ApplicationContext act=new ClassPathXmlApplicationContext("spring-cfg.xml")
    • 区分SpringIOC中Bean的初始化和定义
      • Bean在SpringIOC中是先定义后初始化和依赖注入的
      • SpringIOC中Bean的定义分为三步
        • Resource定位(xml或者是注解定位)
        • BeanDefinition载入(将定位到的信息保存在Bean定义中)
        • BeanDefinition注册,BeanDefinition的信息发布到SpringIOC容器中
      • 注意:即使将BeanDefinition发布到SpringIOC容器中Bean仍未创建实例(仍未初始化),更没有完成依赖注入
      • Bean初始化过程(定义==>初始化==>依赖注入)

Redis简述

  • Redis的性能优越主要是因为

    • 它是基于ANSIC语言编写的,接近于汇编语言的机器语言,运行速度很快
    • redis是基于内存进行io读写,速度比磁盘要快很多
    • redis数据库结构只有6种类型,数据结构比较简单,因此规则比较少,而关系型数据库则是范式,完整性,规范性都要考虑,规则比较多,数据处理业务会比较复杂

  • Redis目前主要使用的场景有:

    • 在javaweb项目中作为缓存来进行使用
      • 缓存常用的数据
        • 实际场景中读操作远远大于写操作,redis直接操作内存
      • 需要告诉读写的场合,需要使用它进行快速读写,关键在于数据的一致性和访问控制
      • 数据库在适用redis作为缓存的时候需要考虑的点
        • 业务数据常用吗,命中率如何,如果命中率很低,就没有必要写入缓存
        • 该业务是读操作多还是写操作多,如果写操作多,也没有必要写入缓存
        • 业务数据大小如何,如果只有几百兆字节的文件会给缓存带来很大的压力,有没有必要
          • 高速读写场景中使用
            高速读写场景中需要使用异步的方式将数据写入数据库
          • 数据缓存到redis,在一定的条件下把缓存中的数据写入到数据库
          • 高并发下的数据安全问题和一致性问题
          • 有效请求和无效请求
          • 事务一致性

  • 如何在java程序中使用redis

    • 先下载jedis.jar
    • jedis的连接池提供了类redis.clients.jedis.Jedispool使用这个对象需要使用类redis.clients.jedis.Jedispoolconfig对连接池进行配置maxIdel,maxTotal,maxWaitMillts
    • java(面向对象)==>Redis(字符串型操作)==>Spring的RedisTemplate(序列化字符串)
    • 使用spring提供的redistemplate之前需要配置spring所提供的连接工厂,在spring-data-redis方案中提供了四种工厂模型
      • JRedisConnectionFactory
      • JedisConnectionFactory
      • LettuceConnectionFactory
      • SrpConnectionFactory
        这四种工厂都是RediscConnectionFactory接口的实现
      • spring中使用redis的时候使用到了序列化器(RedisSerializer)接口

微服务设计原则

发表于 2019-09-14 | 分类于 微服务

微服务设计原则

  • 微服务设计原则

    • 单一职责原则(单一职责是solid原则之一)
    • 服务自治原则(服务自治是指每个微服务应该具备独立的业务能力,依赖与运行环境)
    • 轻量级通信机制(通信机制体量较轻,同时应该是跨语言,跨平台的)
    • 微服务粒度(确定边界,微服务之间应保持独立,并保持松耦合)

  • 微服务注册与发现

    • 服务注册
    • 服务发现
      • 在微服务架构中,服务发现组件是一个非常关键的组件,服务发现组件应该具备的功能有:
        • 服务注册表(记录微服务的服务名称,端口和ip)
        • 服务注册与服务发现
        • 服务检查
      • 各个微服务api的主要作用
        • 查询api用于查询可用的微服务实例,管理api用于服务的注册与注销
        • 服务注册是指微服务启动时,将自己的信息注册到服务发现组件上,服务发现是指用于查询可用微服务列表及其地址的机制
        • 服务发现机制使用一定机制定时检查已经注册了的微服务,如果发现某个实例长时间无法访问,就会从服务注册表中移除该实例

  • 什么是eureka

    • eureka 是netflix开源的服务发现组件,本身是一个基于rest的服务,spring-cloud将它集成在子项目spring-cloud-netflix中,从而实现微服务的注册与发现
    • Region和Zone都是AWS的概念,其中,Region是表示aws中的地理位置,每个Region都有多个availability Zone,多个Region之间完全隔离,AWS就是通过这种方式实现最大的容错和稳定性,Spring-cloud默认使用的Region是us-east-1,在非AWS环境中,可以将Region理解为跨机房的eureka集群
    • Eureka包含有两个组件,分别是Eureka-client和Eureka-server
      • Eureka-server提供服务发现的能力,各个微服务启动时会向Eureka-Server注册自己的信息(比如ip,端口,微服务名称),Eureka-server会存储这些信息
      • Eureka-client是一个java客户端,用于简化Eureka,Eureka-server启动后,Eureka-client会周期性,默认是30s向Eureka-server发送心跳,以续约自己的租期
      • 如果Eureka-server在一定的时间内没有接到一个微服务的心跳,默认是90s,则会注销该实例
      • 默认的情况下Eureka-server也是Eureka-client,多个Eureka-server实例之间通过复制来实现服务注册表中数据的同步
      • Eureka-client会缓存服务注册表中的信息,Eureka通过心跳检查,客户端缓存等机制提高了系统的灵活性,可伸缩性和可用性
